# Тестовое задание для Middle PHP Developer

Это задание основано на реальных паттернах из legacy-кодовой базы: обновление сущности из контроллера, разнородные форматы payload, слабая валидация и хвосты в виде `TODO`/`FIXME`.

## Сценарий

Тебе достался небольшой endpoint, который обновляет данные о продукте в коммерческом предложении.

Сейчас код:

- принимает несколько форматов payload;
- смешивает транспортный слой, валидацию и доменную мутацию;
- обновляет поля сущности через динамическое присваивание;
- обрабатывает relation-поля как отдельные специальные случаи;
- содержит `TODO`/`FIXME`, оформленные в стиле реального проекта.

Задание специально оставлено неаккуратным. Нам важно посмотреть, как ты работаешь с проблемным кодом, который уже выглядит как production-legacy.

## Цель

Улучшить реализацию так, чтобы она стала безопаснее и проще в поддержке, не ломая контракт успешного обновления.

## Ограничения

- Не менять публичные сигнатуры `CommercialProductController::update()` и `CommercialProductController::patch()`.
- Сохранить форму успешного ответа совместимой с описанным контрактом.
- Можно вводить новые классы.
- Можно рефакторить существующие классы.
- Если что-то неоднозначно, можно зафиксировать допущения в отдельном файле.

## Рекомендуемое время выполнения

Рассчитывай примерно на 2-4 часа. Важнее корректность и ход мысли, чем полировка всех возможных edge case.

## На что мы будем смотреть

- Как ты читаешь и упрощаешь legacy-код.
- Сохраняешь ли ты полезное существующее поведение.
- Как ты фиксируешь и объясняешь компромиссы.
- Качество тестов.
- Именование и декомпозицию.

## Как запустить

Локально:

```bash
cd php-middle-test-task
composer install
vendor/bin/phpunit
```

Через Docker:

```bash
cd php-middle-test-task
docker compose run --rm php composer install
docker compose run --rm php vendor/bin/phpunit
```

## Примечания

- Код намеренно неудобный.
- Не каждый `TODO`/`FIXME` обязательно закрывать.
- Небольшие, обоснованные изменения поведения допустимы, если они задокументированы.
